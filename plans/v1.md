# PLANS.md

## Project
Interior Design with Diffusion Models

## Goal
Build a production-ready diffusion-model pipeline for interior design generation and editing in VS Code using Codex.

This plan is the source of truth for implementation order, constraints, acceptance criteria, and delivery milestones.

---

## 1. Product Objective

Create a system that can:
- generate interior design images from text prompts
- redesign an existing room from an input image
- edit selected regions with inpainting
- optionally preserve structure with conditioning such as ControlNet
- expose the workflow through a clean Python project structure and optional demo UI

Primary stack:
- Python
- PyTorch
- Hugging Face `diffusers`
- `transformers`
- `accelerate`
- Gradio for demo UI

---

## 2. Non-Goals

Out of scope for the first version:
- full model training from scratch
- distributed inference infrastructure
- mobile deployment
- full 3D scene generation
- multi-user web platform
- advanced dataset curation pipeline

---

## 3. User Stories

### Core users
- Designer wants quick concept renders from prompts
- User wants to upload a room image and receive redesigned variants
- User wants to replace specific furniture or decor with masked editing
- Developer wants reproducible scripts and a modular codebase

### Success criteria
- User can run text-to-image generation with one command
- User can run image-to-image redesign with one command
- User can run inpainting with one command
- Outputs are saved predictably with metadata and prompts logged
- Code is modular enough to swap base models and pipelines

---

## 4. Functional Requirements

### 4.1 Text-to-image
- Accept prompt and optional negative prompt
- Accept generation parameters:
  - width
  - height
  - guidance scale
  - inference steps
  - seed
  - number of images
- Save outputs to `outputs/text2img/`

### 4.2 Image-to-image
- Accept input image path
- Accept transformation strength
- Preserve overall room composition where possible
- Save outputs to `outputs/img2img/`

### 4.3 Inpainting
- Accept input image path and mask image path
- Replace masked region only
- Save outputs to `outputs/inpaint/`

### 4.4 ControlNet support
- Support at least one structural control mode in first implementation
- Preferred first option: canny or depth
- Save outputs to `outputs/controlnet/`

### 4.5 Prompt management
- Support reusable prompt templates
- Log all prompt inputs with generation parameters
- Maintain example prompt library for:
  - living room
  - bedroom
  - kitchen
  - office
  - bathroom

---

## 5. Non-Functional Requirements

- Reproducible outputs with fixed seed
- Graceful handling of missing GPU
- Clear error messages for invalid inputs
- Consistent file naming convention
- Minimal setup friction for local development
- Code formatted and linted
- Configurable without editing source files directly

---

## 6. Technical Constraints

- Prefer Hugging Face `diffusers` pipelines over custom diffusion implementations
- Default implementation should target local inference first
- GPU acceleration preferred; CPU fallback may be documented but not optimized
- Avoid hardcoding secrets or tokens
- Keep model IDs configurable
- Keep pipeline abstractions thin and testable
